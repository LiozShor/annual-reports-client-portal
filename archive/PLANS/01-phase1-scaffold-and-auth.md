# Phase 1: Project Scaffold + Auth Endpoints

**Status:** COMPLETED (session 172, DL-169)

## Goal

Stand up the Cloudflare Workers project, prove the stack works end-to-end, and migrate the two simplest endpoints: **login** and **token verification**. These are pure crypto (HMAC-SHA256) with zero Airtable data queries, making them the ideal first migration.

---

## Endpoints Being Migrated

### 1. `POST /admin-auth` (Login)

**Current workflow:** `[Admin] Auth & Verify` (REInXxiZ-O6cxvldci3co)
**Webhook path:** `/admin-auth`

**Current logic (from n8n Code node "Validate & Generate Token"):**
1. Receive `{ password }` in POST body
2. Compare against `ADMIN_PASSWORD = 'reports3737!'`
3. On failure: return `{ok: false, error: 'Invalid password'}`, log security event
4. On success:
   - Build payload: `{exp: Date.now() + 8*60*60*1000, iat: Date.now(), type: 'admin'}`
   - Base64-encode payload → compute HMAC-SHA256 signature with `SECRET_KEY`
   - Token format: `base64(JSON_payload).hex(HMAC_signature)`
   - Log AUTH_SUCCESS to Airtable security_logs
   - Return `{ok: true, token: token}`

**Security logging (fire-and-forget):**
- Direct HTTP POST to `https://api.airtable.com/v0/appqBL5RWQN9cPOyh/security_logs`
- Fields: `timestamp`, `event_type`, `severity`, `actor`, `actor_ip`, `endpoint`, `http_status`, `error_message`, `details`, `workflow_execution_id`
- Uses separate Airtable PAT (`pat2XQGRyzPdycQWr...`)
- Failures silently caught (never block the response)

### 2. `GET /admin-verify` (Token Verification)

**Current workflow:** Same as above
**Webhook path:** `/admin-verify`

**Current logic (from n8n Code node "Verify Token"):**
1. Extract token from `Authorization: Bearer {token}` header OR `?token=` query param
2. Split token on `.` → `[dataB64, signature]`
3. Recompute HMAC-SHA256 of `dataB64` using `SECRET_KEY`
4. Compare signatures (hex string comparison)
5. Decode dataB64 → parse JSON → check `exp >= Date.now()`
6. Return `{ok: true/false}` (with `error` on failure)
7. Log TOKEN_EXPIRED or TOKEN_INVALID on failure

### Key Implementation Details

#### Token Module (`lib/token.ts`)
Port the exact HMAC-SHA256 logic from n8n. Critical: use `crypto.subtle` (Web Crypto API available in Workers) instead of Node.js `crypto`.

```
Token format: base64url(JSON_payload).hex(HMAC_SHA256_signature)
Payload: {exp: ms_timestamp, iat: ms_timestamp, type: 'admin'}
Secret: ADMIN_SECRET_KEY (Worker secret)
Expiry: 8 hours from creation
```

**Cross-compatibility requirement:** Tokens generated by the Worker must be verifiable by n8n Code nodes (and vice versa) during the migration period. This means:
- Same signing key
- Same base64 encoding (standard, not URL-safe — match n8n's `Buffer.from().toString('base64')`)
- Same HMAC hex output format

#### CORS Middleware (`middleware/cors.ts`)
```
Access-Control-Allow-Origin: https://liozshor.github.io
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
```
Must handle OPTIONS preflight automatically. Hono has built-in CORS middleware that covers this.

#### Auth Middleware (`middleware/auth.ts`)
Reusable middleware that:
1. Extracts Bearer token from Authorization header (or `?token=` query param fallback)
2. Verifies HMAC signature
3. Checks expiry
4. Sets `c.set('tokenPayload', payload)` for downstream handlers
5. Returns 401 JSON on failure

This middleware is used by every protected route in Phases 2-5.

#### Client Auth (`middleware/client-auth.ts`)
Separate middleware for client-facing endpoints (Check Existing Submission, Reset Submission, Get Client Documents):
- Different secret key: `CLIENT_SECRET_KEY`
- Different token format: `expiryUnix.hex(HMAC_SHA256)`
- 45-day expiry
- Timing-safe comparison (`crypto.subtle.timingSafeEqual`)

#### Airtable Client (`lib/airtable.ts`)
Generic, typed REST client for Airtable — built in Phase 1, used by all later phases:
- `search(tableId, filterByFormula, fields?, sort?)` → paginated (handles `offset`)
- `get(tableId, recordId)` → single record
- `create(tableId, fields)` → single create
- `createBatch(tableId, records[])` → batch create (max 10 per call)
- `update(tableId, recordId, fields)` → single update
- `updateBatch(tableId, records[])` → batch update
- `deleteRecord(tableId, recordId)` → single delete
- `deleteBatch(tableId, recordIds[])` → batch delete (max 10)
- Automatic pagination for search (follows `offset` token)
- Configurable base ID (default: `appqBL5RWQN9cPOyh`)
- Bearer token from Worker secrets

#### Security Logger (`lib/security-log.ts`)
Fire-and-forget function that POSTs to Airtable `security_logs` table:
```typescript
function logSecurityEvent(env: Env, ctx: ExecutionContext, event: SecurityEvent): void {
  ctx.waitUntil(
    fetch(`https://api.airtable.com/v0/${BASE_ID}/security_logs`, {
      method: 'POST',
      headers: { Authorization: `Bearer ${env.AIRTABLE_PAT_SECURITY}`, 'Content-Type': 'application/json' },
      body: JSON.stringify({ fields: { ...event } })
    }).catch(() => {}) // Silent failure — never block response
  );
}
```
Uses `ctx.waitUntil()` so the log completes after the response is sent, adding zero latency.

## Acceptance Criteria

1. **Deploy Worker** to Cloudflare (initially on `*.workers.dev` subdomain)
2. **Login test:** POST to Worker `/admin-auth` with correct password → returns valid token
3. **Verify test:** GET Worker `/admin-verify` with that token → returns `{ok: true}`
4. **Cross-compatibility test:**
   - Generate token via Worker → verify via n8n Code node → passes
   - Generate token via n8n → verify via Worker → passes
5. **CORS test:** Browser on `liozshor.github.io` can call Worker endpoints without CORS errors
6. **Security logging:** Auth attempts appear in Airtable security_logs table
7. **Frontend integration:** Update `ADMIN_AUTH` and `ADMIN_VERIFY` URLs in `shared/endpoints.js` → login works identically in the admin portal
