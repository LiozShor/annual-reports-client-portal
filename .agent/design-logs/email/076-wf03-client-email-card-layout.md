**Status:** [IMPLEMENTED]
**Date:** 2026-03-02
**Related Logs:** 030-bilingual-client-email.md, 071-bilingual-doc-list-all-emails.md, 073-type-a-single-cta-mirror-wf01.md

## 1. Context & Problem

The Type B reminder email (WF[06]) uses a polished card-based bilingual layout: each language gets its own bordered card with a language tag (`🔤 English` / `🔤 עברית`), distinct backgrounds (white for EN, `#f9fafb` for HE), and a shared bilingual footer. The Batch Status email already matches this pattern.

The WF[03] client email (generated by Document Service "Generate HTML" node) still uses the older separator-based layout: a dashed line with "גרסה בעברית / Hebrew version" label between flat tables. No cards, no language tags, no distinct backgrounds.

**Goal:** Update the WF[03] bilingual client email to match the Type B card pattern for visual consistency across all bilingual emails.

## 2. User Requirements

1. **Q:** Header bar style? (Type B uses tone-colored header)
   **A:** Static blue header bar — always blue (`#2563eb` bg, white text), like Type B's "friendly" tone.

2. **Q:** Keep per-card instructions (send docs to email)?
   **A:** Yes — each card is self-contained. Keep instructions in both language sections.

3. **Q:** Update the Hebrew-only path too?
   **A:** No — Hebrew-only path stays as-is.

## 3. Research

### Domain
Bilingual transactional email design, card container patterns, RTL/LTR layout.
Prior research: DL-030, DL-071 (bilingual email structure, RTL handling, doc list rendering).

### Sources Consulted (Incremental)
1. **Flodesk — Bilingual Email Templates** — Card-separated sections are the standard for bilingual emails; clear visual boundaries reduce cognitive load.
2. **RTL Styling 101 (rtlstyling.com)** — `dir` attribute per section is critical; never set globally on bilingual content.
3. **Litmus — Responsive Email Stacking** — Bordered card sections in tables use `border` on `<table>` or wrapping `<td>`, both render reliably.

### Key Principles
- **Visual containment** — bordered cards create clear "this is one language" boundaries
- **Self-contained sections** — reader only needs to read their card, gets all info
- **Consistent pattern recognition** — same card layout across all emails trains users

### Research Verdict
Straightforward visual alignment — apply the existing proven Type B pattern to WF[03]. No new patterns needed.

### Changes Made

### Implementation Deviations from Blueprint
- Blueprint used `<p>` tags with `margin` for card content; implementation uses `<table>/<tr>/<td>` with `padding` (consistent with rest of the code and email-design-rules.md anti-`margin` rule).
- Blueprint used CSS shorthand `padding:24px 32px`; implementation uses longhand `padding-top:24px; padding-right:32px; padding-bottom:24px; padding-left:32px;` per email design rules.
- Blueprint had `<h1>` in header bar; implementation uses `<table><tr><td>` with inline styles for Outlook compatibility.
- Added `\u{1F524}` (🔤 emoji) via ES6 Unicode escape for the non-BMP character.
