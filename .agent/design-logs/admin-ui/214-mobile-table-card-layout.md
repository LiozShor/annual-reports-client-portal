# Design Log 214: Mobile Table → Card Layout for Admin Portal
**Status:** [COMPLETE — TESTED 2026-03-28]
**Date:** 2026-03-28
**Related Logs:** [193-mobile-responsiveness-audit](../infrastructure/193-mobile-responsiveness-audit.md), [212-mobile-bottom-nav-ai-review](212-mobile-bottom-nav-ai-review.md), [087-responsive-floating-elements](087-responsive-floating-elements.md)

## 1. Context & Problem
All 5 admin portal tables (Clients, Pending, Review Queue, Reminders, Questionnaires) render as horizontal tables on mobile, forcing horizontal scroll. Status badges, action buttons, and data columns compete for space. The search/filter header area consumes ~40% of the viewport before any data appears.

DL-193 audited mobile issues but focused on emails and basic CSS fixes (padding, font sizes). DL-212 added bottom nav. Neither addressed the fundamental table layout problem — tables need to become cards on mobile.

## 2. User Requirements
1. **Q:** Which tables get the mobile card treatment?
   **A:** All 5 tables — Clients, Pending, Review Queue, Reminders, AND Questionnaires

2. **Q:** How should card actions (print, folder, expand, checkboxes) be handled?
   **A:** Actions row at the bottom of each card — always visible, no gesture needed

3. **Q:** Should mobile cards support bulk selection?
   **A:** Yes, always show checkbox on each card (no long-press gesture)

4. **Q:** Should the search/filter header be collapsible on mobile?
   **A:** Yes — sticky collapsed bar with search icon + active filter count. Tap to expand full filters.

## 3. Research
### Domain
Responsive Data Tables, Mobile Card Layouts, RTL Responsive Design

### Sources Consulted
1. **Prior research (DL-193, 2026-03-26)** — Fluid hybrid patterns, touch targets (44px min), RTL auto-reversal with Flexbox/Grid, iOS auto-zoom prevention at 16px
2. **CSS-Tricks / Adrian Roselli** — `display: block` on `<table>` destroys screen reader semantics in Safari (completely) and partially in other browsers. Adding ARIA roles back is incomplete. Dual-render approach (native `<table>` on desktop, `<ul>/<li>` cards on mobile) is the accessible solution.
3. **Smashing Magazine / UXmatters** — Admin panel rows are independent records (not comparison data), making cards the natural mobile format. Card layout should show: primary info (name, status), secondary info (counts, dates), and actions.
4. **Material Design 3 / Apple HIG** — Card minimum touch targets: 44x44px. Bottom action row pattern well-established. Checkbox placement: leading edge (start of card in reading direction).

### Key Principles Extracted
- **Dual render, not CSS transform**: Generate both `<tr>` and card markup per record. Toggle visibility with CSS `@media`. Avoids accessibility destruction from `display: block` on `<table>`.
- **Progressive disclosure on cards**: Primary info (name + status) always visible. Secondary info (email, dates, counts) shown below. Actions in dedicated bottom row.
- **RTL-safe CSS**: Use logical properties (`margin-inline-start`, `inset-inline-start`) for all card styles. Flexbox auto-reverses in `dir="rtl"`.
- **Touch targets**: All card action buttons and checkboxes minimum 44x44px.
- **Filter collapse**: Sticky bar with filter count badge. Expand/collapse with slide animation. Applied filter count gives users confidence their filters are still active.

### Patterns to Use
- **Dual render with CSS toggle**: `<table>` visible at `>768px`, `.mobile-card-list` visible at `≤768px`. Both generated by the same JS render function.
- **Card anatomy**: Checkbox | Primary line (name + stage badge) | Secondary line (docs, date, email) | Actions row (existing buttons)
- **Collapsible filter bar**: `.filters-mobile-collapsed` (sticky, thin) ↔ `.filters-mobile-expanded` (full filters). Toggle with JS class swap.

### Anti-Patterns to Avoid
- **CSS `display: block/flex` on `<table>/<tr>/<td>`**: Destroys accessibility tree. Tempting because it's less code, but breaks screen readers.
- **JS `matchMedia` for DOM swaps**: Adds complexity, potential race conditions on resize. CSS `display: none` is sufficient and removes from render tree.
- **Hiding all secondary info on cards**: Tempting to show only name + status, but users need docs count and dates without tapping into each card.

### Research Verdict
Dual-render approach: each table render function generates both a `<table>` (desktop) and a `<ul class="mobile-card-list">` (mobile). CSS media queries toggle visibility. Cards use the anatomy pattern above. Filter bar gets a collapsible sticky wrapper on mobile. This adds ~30% more DOM but at admin scale (20-50 rows) it's negligible.

## 6. Proposed Solution (The Blueprint)
### Logic Flow
1. **New CSS card classes** — Define `.mobile-card-list`, `.mobile-card`, `.mobile-card-primary`, `.mobile-card-secondary`, `.mobile-card-actions` with mobile-only visibility
2. **Modify 5 render functions** — Each generates BOTH `<table>` AND `<ul class="mobile-card-list">` markup. Insert card list after table inside `.table-scroll-container`.
3. **CSS media query toggle** — At `≤768px`: hide `<table>`, show `.mobile-card-list`. At `>768px`: show `<table>`, hide `.mobile-card-list`.
4. **Collapsible filter bar** — Add `.filters-collapsed` wrapper visible only at ≤768px. Full `.filters` hidden at ≤768px by default, toggled via JS.
5. **Card anatomy per table:**

| Table | Primary Line | Secondary Line | Actions |
|-------|-------------|---------------|---------|
| Clients | Name + Stage badge | Docs progress + Missing count + Notes (truncated) | Send/Reminder btn, overflow menu |
| Pending | Name | Email + copy btn | (checkbox only) |
| Review | #N + Name + Waiting badge | Year + Docs + Completion date + Email | View, Mark complete |
| Reminders | Name + Status badge | Next date + Last sent + Count/Max + Docs progress | Send btn |
| Questionnaires | Name + Stage badge | Spouse + Submission date | Open docs, Print, Expand |

### Data Structures / Schema Changes
None — purely UI rendering changes.
